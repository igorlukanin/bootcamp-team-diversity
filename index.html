Различия команд в Управлении разработки

Игорь Луканин, сентябрь 2017

~50 команд

Сходства и различия

  • Люди
  • Процессы
  • Продукты
  • Технологии

Люди

  • Размер команды
  • Возможности для роста (как тимлид, как техлид)
  • Привычки в общении (шум, лексика)

Процессы

  • Итеративность разработки (скрам, канбан, проекты)
  • Распределённость команды (один или несколько городов)
  • Количество продуктов (один или несколько)
  • Работа в парах
  • Фулстек-задачи
  • Коллективное владение кодом или специализация
  • Кто пишет код (разработчики, тестеры, аналитики)

Продукты

  • Аудитория продуктов (внешняя или внутренняя)
    • физические лица, малый и крупный бизнес, государство
    • работники, другие команды
  • Зрелость продуктов (активные, на старте, на поддержке)
  • Количество пользователей
  • Сезонность нагрузки
  • Наличие дедлайнов (законы, зависимости от других команд)
  • Наличие SLA, интеграций и публичного API

Общие технологии

  • Микросервисная архитектура
  • Отказоустойчивые топологии
  • Автоматизированное тестирование
  • Непрерывная интеграция
  • Интеграция с общей инфраструктурой и другими продуктами

Технологии

  • Платформы (backend, web, mobile, desktop, API)
  • Типы нагрузки (много запросов, данных, взаимосвязей)
  • Источники данных (пользователи, внешние источники)
  • Языки программирования и другие технологии
    • C#, CIL, С++, Java, Kotlin, Scala, R, Python, Ruby, Node.js, Go, 1С, VBA
    • ECMAScript 5–6, TypeScript, Flow, React, Angular, jQuery, IE 8+

Технологии, продолжение

  • Хранилища данных
    • MS SQL Server, MariaDB
    • MongoDB, Cassandra, ClickHouse, Zebra
    • Redis, HDFS, Ceph, Kanso
  • Поисковые хранилища и движки (Elasticsearch, Sphinx)
  • Хранилица конфигурации (ZooKeeper, Consul, ClusterConfig)
  • Очереди сообщений (RabbitMQ, Kafka)
